How many people can one GPU serve?

Everyone benchmarks a local model with one prompt and one stream. It is the wrong test for anything real. Here is what actually happens under concurrency, with real numbers.

Quick answer: on our bench, one GPU running gpt-oss-120b went from ~160 tokens/sec at a single user to ~310 at eight users - nearly double the total work from the same hardware - while each user's speed dropped from ~171 to ~79. More users means more total tokens, but a slower turn for everyone. Batching does not make a GPU faster; it makes it fuller.

If you have ever looked up "how fast is model X locally," you got a single-stream number: one prompt, one reply, tokens per second. That number is real, but it describes a situation almost no real deployment is in. The moment your GPU answers more than one request at a time, the picture changes completely - and almost nobody measures that. So we did.

§1

Why is the single-stream benchmark misleading?

A single stream leaves most of a modern GPU idle between tokens. The card is waiting on memory far more than it is computing, so one user never keeps it busy. The single-stream tokens-per-second figure is the best case for one person, not the ceiling of the hardware. Quote it and you will badly underestimate what the box can actually produce.

§2

What happens when you serve more than one person?

We ran gpt-oss-120b on a 4x RTX PRO 4500 rig and swept the number of concurrent users from 1 to 8. Two things move at once, in opposite directions:

  • Aggregate throughput climbs. ~160 tokens/sec at one user, ~310 at eight. Nearly double the total work out of the exact same hardware.
  • Per-user speed drops. That single user had ~171 tokens/sec all to themselves. By eight concurrent users, each one is down to ~79. Less than half.

That is the whole trade in two lines: more total tokens, a slower turn for everyone.

§3

Why does total throughput go up while each user slows down?

It is called continuous batching - what runtimes like vLLM do under the hood. Instead of finishing one request before starting the next, the GPU interleaves many requests, filling the idle gaps between one user's tokens with another user's work. The expensive silicon stops waiting around, so the aggregate line rises. But that shared capacity is now split across everyone, so any single user waits a little longer for their turn.

§4

Is a single-user home rig wasteful, then?

On pure throughput, yes. If you run a local model only for yourself, you are using a fraction of what the GPU can produce. That ~160 tok/s at one user is not the ceiling - the ceiling was ~310, and you leave most of it on the floor because you are the only one asking. A single-user rig is, by definition, the least efficient way to run inference hardware: you bought a machine that can serve eight people and you are serving one.

§5

When does the batching trade make sense?

It depends on whether you are latency-bound or throughput-bound. For a queue or a background job that tolerates a 10-45 second turn - overnight processing, a chatbot for a group, batch extraction - batching is a clear win: you clear far more work per hour. For a real-time voice assistant that needs the fastest possible single reply, concurrency hurts, and you want the single-stream speed instead. Know which one you are.

§6

How do you get more requests than one person can generate?

This is the quiet argument for pooling. A GPU serving one household wastes most of its throughput; the same GPU fed a steady stream of requests from many people runs near its real capacity, and every one of those extra tokens costs almost nothing because the hardware and power were already committed. §7 · FAQ

Frequently asked questions

How many concurrent users can one GPU handle?
It depends on the model and VRAM, but aggregate throughput keeps rising until compute or memory bandwidth saturates. On our bench a single 4x RTX PRO 4500 rig running gpt-oss-120b nearly doubled its total tokens/sec going from 1 to 8 concurrent users, while per-user speed more than halved.
Does serving more users make a GPU faster?
No. Batching does not speed the GPU up - it keeps it fuller. Total throughput rises because idle gaps between one request's tokens get filled with another request's work. Each individual user actually gets a slower turn.
Why is my local model slow with only one user?
A single stream leaves the GPU idle between tokens, so you only see a fraction of its real throughput. That is expected: one user cannot keep a capable GPU busy. The capacity is there, waiting for more concurrent requests than one person generates.
Should I benchmark local serving with one stream?
Only if you truly serve one user at a time. For anything with a queue or multiple users, measure aggregate throughput under concurrency - that is what determines how much work the box can actually do.
What is continuous batching?
A serving technique (used by vLLM and others) that interleaves many requests so the GPU's compute stays saturated, instead of finishing one request before starting the next. It is why aggregate throughput scales with concurrency.
