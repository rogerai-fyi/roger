RogerAI - the operating manual.

A two-way radio for GPUs. Borrow a model from someone's home GPU, or lend out your own and get paid per token. This is the field manual: how to install the set, tune in, go on air, run private channels, settle the wallet, and check the lineage on every token. Read it once, keep it by the dial.

Set: rogerai Version: v4.8.2 Broker: broker.rogerai.fyi Protocol: OpenAI-compatible
Quickstart - the 30-second path

Three commands, no account.

Install the set, then tune in to a model or put your GPU on air. Browsing the band and opening a free endpoint need no login - you only sign in to spend from a wallet or to earn as a provider.

macOS or Linux, one line (Windows: install.ps1 in PowerShell).

$ curl -fsSL https://rogerai.fyi/install.sh | sh

Free, no login: the radio detects your local model and puts it on air.

$ roger onboard --free   # set up to share, no login

§0 · Front matter

What this set is

RogerAI is a two-way radio for GPUs. Powerful open models sit idle on home GPUs everywhere. RogerAI opens that silicon as one shared band: an operator puts a model on air at a price they set; you tune in and pay per token. Every token carries a lineage receipt. One small CLI does both sides - there is nothing else to run.

You never run a server, open a port, or sign up to browse. To consume you install the binary and tune in; the radio hands you a local OpenAI-compatible endpoint on 127.0.0.1 and bills your wallet per token. To provide, you point the same binary at a model you already run (Ollama, LM Studio, llama.cpp, vLLM, LiteLLM) and flip the switch. Free by default, no login; set a price and link a GitHub account to earn.

Glossary

band
A model. Every model id (for example qwen3-coder-30b) is a band you can tune to. More stations on a band = stronger signal.
station
A provider node - one home GPU serving a band. Owners set their own price; the broker ranks them.
channel
Your live connection to a station: a local endpoint + key the radio opens for you, plus the in-app chat session.
on air
Serving. A station that is on air is advertised in the band and can take traffic.
tune in
Consuming. You pick a band, set margins, and the radio locks the strongest station.
lineage receipt
The signed record co-produced for every request: model, shard, token counts, price - node-signed and broker counter-signed (Ed25519). See §10.
grant key
An owner-issued private credential (rog-grant_...) that lets your bots, family, or your own remote box use your models with no login - free or at a price you sponsor. See §5.
verified operator
The ✓ mark of a GitHub-linked operator that co-signs each token's lineage. It is an IDENTITY badge - it does NOT by itself mean the node is confidential.
confidential ◆
The gold ◆ is granted ONLY to a node that passes real TEE remote attestation: the broker cryptographically verifies a hardware quote (AMD SEV-SNP signature chain to the AMD root, a nonce binding that prevents replay, and an allowlisted launch measurement) and re-checks it on a cadence. Nodes without it are plainly "standard". See §4 and §10.
§1 · Installation

Install the set

One command on macOS or Linux. It downloads the right binary for your OS/arch from GitHub Releases, verifies its SHA256 against the published checksums, and installs it to ~/.local/bin.

$ curl -fsSL https://rogerai.fyi/install.sh | sh

On Windows, in PowerShell:

PS> irm https://rogerai.fyi/install.ps1 | iex
FIG. 1 - PLATFORMS
Linuxamd64 · arm64 - installs to ~/.local/bin (override with ROGERAI_INSTALL_DIR)
macOSamd64 (Intel) · arm64 (Apple silicon)
Windowsamd64 · arm64 via the PowerShell installer (install.ps1)

Upgrade in place

The binary self-updates. It downloads the latest release, verifies the checksum, and atomically swaps the running file.

$ roger upgrade
$ roger upgrade --check   # just report whether a newer version exists

A subtle, cached update banner appears at startup when a newer release is out. Disable it with ROGERAI_NO_UPDATE_CHECK=1. Re-running the installer also upgrades in place.

Build from source

The whole client is one Go binary in the public repo. With a Go toolchain:

$ git clone https://github.com/rogerai-fyi/roger
$ cd roger
$ go build -o rogerai ./cmd/rogerai
§2 · First contact

The first run

Run rogerai with no arguments. On a real terminal, a tiny first-run wizard asks the one question that matters - are you here to use models or to share your GPU - then drops you into the app. Everything else (the model, the context length, a free local port) is auto-detected. The wizard runs once; re-run it any time with roger onboard.

$ rogerai
  Welcome to RogerAI - a two-way radio for GPUs.
  Are you here to use models, or to share your GPU?
   > Just use models (open the app)
     Share my GPU - QuickStart, FREE, no login
     Share my GPU - earn (set prices + log in)

Share for free in 30 seconds

Pick the FREE share path (or run it straight): no login, no account. The radio detects your local model and puts it on air.

$ roger onboard --free   # non-interactive: set up to share, no login
$ roger share             # go on air with the detected model

Use a model in 30 seconds

Choose to use models and you land in the app: browse the band, tune in, chat. Or skip the app and go straight to an endpoint for your bots:

$ roger use qwen3-coder-30b

§A · The [0] agent

An agent that runs on the band

Press 0 in the app to open [0] AGENT: a small tool-using agent that runs on whatever model you have tuned in. Ask it to do something and it loops - it can read files, list directories, and fetch the web on its own; writing files and running shell commands always stop to ask you first. It is session-only: it keeps no memory between runs.

The model it uses

The agent never reaches for a stale default. It runs on, in order: your open channel if you have one, else the last band you tuned in this session, else it asks you with /model. With no model tuned in it tells you so and points you at 1 to tune in. If the tuned model cannot make tool calls, the agent degrades to a plain chat rather than failing.

The tools, and what asks first

The tool set is small and bounded. Read-only tools run on their own; anything that changes your machine is confirm-gated (y / N, default no). File tools are sandboxed to the directory you launched in - absolute paths and ../ escapes are rejected.

read_fileread a file under the working dir - runs on its own
list_dirlist a directory under the working dir - runs on its own
web_fetchfetch a URL - runs on its own
write_filewrite a file (sandboxed) - asks first (y/N)
run_shellrun a shell command in the working dir - asks first (y/N); runs in the cwd but is not sandboxed

Each tool call shows a short preview of what it did, so you can follow the agent's work as it goes.

The persona

The agent's voice and house rules come from a plain Markdown persona at ~/.config/rogerai/dj.md, written on first run and yours to edit. /persona prints its path so you can open and tune it.

§3 · Tune in (consume)

Lock a signal, get an endpoint

There are two ways to consume: the interactive app (the dial) for browsing and chatting, and roger use for a headless local endpoint your tools talk to.

The app (the dial)

Run rogerai to open the app and press 1 for TUNE IN: the live band, sorted strongest first. Move the cursor with ↑/↓ (or j/k), Enter to tune in. The list is virtualized, so a crowded band stays fast. When you are connected, Enter on that row jumps into the chat, and d disconnects right from the list.

Headless: roger use

This is the drop-in for Hermes, your bots, or any tool that speaks OpenAI. Pick a band, set margins, and the broker locks the strongest station and hands you a local endpoint + key. When a station fades, it re-tunes.

$ roger use qwen3-coder-30b --max-out 0.30
$ roger use qwen3-coder-30b --max-out 0.40 --advanced --min-tps 40 --port 4141
FIG. 2 - ROUTING & PORTS

Portauto-picked free port from 4141 up; pin it with --port
Routingbroker ranks stations by price then measured speed; --max-out is the headline cap
Margins--max-out $/1M out (headline) · --max-in · --min-tps floor (advanced)
Billingper token, debited live from your wallet balance in $

Pay per token, watch it tick

Prices are set by each station, in dollars per 1M tokens (in and out); typical bands run roughly $0.18-$0.55 / 1M. A typical chat reply costs a fraction of a cent. Your wallet balance is shown in the app header and via roger balance, and it ticks down live as you generate.

Resilient by design: transparent re-routing

You tune to a band (a model), not to one station. The broker hands you the best available station for that model, and the client wraps every request in a bounded failover loop: if a station times out, drops the connection, or returns a 5xx mid-stream, RogerAI automatically re-routes to another station serving the same model - a few attempts with exponential backoff (200ms, capped at 2s), each picking the next fast-and-cheap station. A 4xx (your own bad request) is never retried - that is not the station's fault. Your local OpenAI endpoint stays fixed throughout, so your bots and tools see one stable endpoint and never notice a station went off air. If every station on the band is exhausted you get one clear error, not a silent hang.

Many models at once: two sets, or call the broker direct

One roger use is one set tuned to one channel. It opens a single local endpoint (default http://127.0.0.1:4141/v1) and that endpoint always speaks to the one band you tuned. Re-tuning the same set to another band reuses that port and drops the first channel - so a single rogerai process never serves two models at once (in the app it is the same: the open channel is singular - tuning a new band replaces it). To consume several models simultaneously, pick one of two ways.

(a) Run a second set on its own port. Each instance binds its own local endpoint, so two models live side by side. Pin the port with --port so the URLs are stable for your tools:

# terminal 1 - a coder on 4141
$ roger use qwen3-coder-30b --port 4141   # http://127.0.0.1:4141/v1

# terminal 2 - a general model on 4142, at the same time
$ roger use gpt-oss-120b --port 4142      # http://127.0.0.1:4142/v1

Point one tool at :4141 and another at :4142 and both bands run concurrently, each with its own spend caps. This is the simplest path when you want a couple of fixed endpoints. (Each set is one blocking process - keep both running, one per model.)

(b) Skip the local proxy - point straight at the broker. The cleaner path when you want many models from one endpoint: with a grant key (§5), aim any OpenAI client at the broker's own /v1 and switch models by setting the model field per request. No local proxy, no per-model process, no ports to juggle - the broker routes each request to a station for whatever model you named.

$ export OPENAI_API_BASE=https://broker.rogerai.fyi/v1
$ export OPENAI_API_KEY=rog-grant_3f9a...c10b   # a grant key - see §5

# same endpoint, same key, different model per call - all at once
$ curl https://broker.rogerai.fyi/v1/chat/completions \
    -H "Authorization: Bearer $OPENAI_API_KEY" \
    -d '{"model":"qwen3-coder-30b","messages":[{"role":"user","content":"hi"}]}'
$ curl https://broker.rogerai.fyi/v1/chat/completions \
    -H "Authorization: Bearer $OPENAI_API_KEY" \
    -d '{"model":"gpt-oss-120b","messages":[{"role":"user","content":"hi"}]}'
(a) second setone extra roger use --port per model; fixed local URLs; per-model spend caps; no login or grant needed for free bands
(b) broker directone endpoint serves every model; set model per request; needs a grant key (§5) but no local proxy at all - the right call for a fleet of bots
note Path (b) routes a grant only to your own nodes. To consume the open public market across many models from one endpoint, use a funded wallet via the local proxy (path a). See §5 for minting a grant and §6 for the wallet.
§4 · Go on air (provide)

Put your GPU on the air

Point RogerAI at a local backend you already run. It auto-detects Ollama, LM Studio, llama.cpp, vLLM, LiteLLM, and anything that serves /v1/models - including a model on a custom port. Then it bridges only that model port over an outbound long-poll: no inbound ports, no shell, no files, no context.

$ roger share                 # FREE by default, no login - auto-detects your model
$ roger share --model qwen3-coder-30b --price-out 0.30
$ roger share --upstream http://127.0.0.1:8081/v1 --model my-model

Free by default, no login

A bare roger share goes on air at price 0/0 with no account. Plenty of stations broadcast for the love of the band. You appear in the band within about ten seconds.

free path No login is required to share for free. Login (a linked GitHub account) is required only once you set a price - a priced node without a login is refused by the broker.

Earn: set a price

Set a per-token rate to earn. Prices are in $/1M tokens, input and output, and you keep 70% of every token you serve (70/30 in the operator's favor). Earning needs a linked GitHub: roger login, then share.

$ roger login                 # link GitHub (device flow) - only to earn
$ roger share --price-in 0.18 --price-out 0.30

Two price guards keep you honest by accident: if your out-price is more than 3× the live market median for that model, share prints a non-blocking "double-check it's not a typo" heads-up before going on air; and the broker hard-refuses to list a public node above $100 / 1M out or $50 / 1M in (a real model never costs that, so only a fat-finger trips it). If you genuinely want an unreachable price, that is what a private band is for (§5).

Self-healing: stays on air across broker restarts

A share is durable. The broker persists its node registry, so a broker restart or redeploy no longer drops a running provider - it re-hydrates the registry and your station keeps serving without you touching anything. If the broker briefly goes away, the provider single-flights one quiet re-registration (a token refresh) and rides it out. The SHARE panel tells the truth about which state you are in: ON AIR when the broker is acknowledging your heartbeats, RECONNECTING while they are failing - never a green light over a dead link.

Schedules and concurrency

--free-windowa daily FREE window in UTC, e.g. 03:00-03:30
--scheduletime-of-use pricing as JSON, e.g. [{"start":"18:00","end":"22:00","price_in":0.5,"price_out":0.7}]
--parallelconcurrent poll workers per node (max-concurrent); default 4
--ctxcontext length; auto-detected from the upstream, falls back to 32768

The confidential tier

Run roger share --confidential on real TEE hardware (today: an AMD SEV-SNP confidential VM). At registration the node generates a fresh hardware attestation quote bound to its key and a broker-issued nonce; the broker cryptographically verifies it - the SEV-SNP signature chain up to the AMD root, the nonce binding (so a quote cannot be replayed), and an allowlisted launch measurement - and re-checks it on a cadence. Only a node that passes carries the gold ◆; a node with no TEE produces no quote, makes no claim, and is plainly "standard". --confidential fails clearly if no TEE is present rather than sending a fake claim. Consumers reach this tier with --confidential on use or /confidential in the app, which routes only to genuinely-verified nodes.

Most provider flags live behind --advanced: roger share --advanced reveals --node, --region, --parallel, --upstream, --ctx, --confidential, --free-window, and --schedule.

§5 · Private channels (grant keys)

Free, private access for your own people

A grant key is a second, private tier on the same node: an owner-issued, labeled bearer credential (rog-grant_...) that your bots, family, friends, or your own remote box use as their API key - with no login, no account, no wallet. The broker resolves the grant to you, routes only to your nodes, applies the grant's price (free or a rate you sponsor), and enforces its caps.

$ roger grant create --name my-bots                   # a FREE key for your bots/family
$ roger grant create --name jane --price-out 0.30     # a priced key you sponsor
$ roger grant create --self --name hermes-box         # a $0 key for YOUR own remote box
the secret is shown once The grant secret prints exactly once at create time - copy it then. The dashboard stores only its hash. Lost it? You cannot recover it - mint a new key and revoke the old one.

Prefer a screen? Mint, list, edit caps and revoke keys on the web API keys page - same endpoints, same one-time secret.

Headless and bots: how a key is used

A grant key is an OpenAI-style API key. There is nothing to install on the consuming side and no login: a headless bot points an OpenAI-compatible client at the broker's /v1 endpoint and sends the grant secret as its bearer token. The broker recognises the rog-grant_ prefix, resolves the key to you, routes only to your nodes, and bills per the grant (free, $0 self, or your sponsored price). The two lines printed at create are all a bot needs:

# printed once by `grant create` (and on the web keys page)
OPENAI_API_BASE=https://broker.rogerai.fyi/v1
OPENAI_API_KEY=rog-grant_3f9a...c10b

That is the whole mechanism: set those two env vars (or the equivalent base_url + api_key in your SDK) and every POST /v1/chat/completions the bot makes is authenticated by the key alone. No roger login, no wallet, no signed identity on the bot box. A worked example - a Petlings or Minecraft chat bot calling your home GPU from a cheap VPS:

# 1) you, once, on your logged-in machine: mint a capped key
$ roger grant create --name petlings-bot --self --monthly-cap 20000000
  created grant "petlings-bot"  (self ($0 on your own boxes))
  rog-grant_3f9a...c10b
  save it now - it is shown only once.

# 2) on the bot box (anywhere): set the two lines and run your bot
$ export OPENAI_API_BASE=https://broker.rogerai.fyi/v1
$ export OPENAI_API_KEY=rog-grant_3f9a...c10b
$ python petlings_bot.py        # any OpenAI client - openai-python, curl, llama-index...
# the raw call your bot makes - just a bearer token
$ curl https://broker.rogerai.fyi/v1/chat/completions \
    -H "Authorization: Bearer $OPENAI_API_KEY" \
    -H "Content-Type: application/json" \
    -d '{"model":"qwen3-coder-30b","messages":[{"role":"user","content":"hi"}]}'

Use --self (above) when the bot calls your own GPU - it is metering-only, $0. Drop --self for a plain free key you hand a friend, or set --price-out to sponsor a paid key billed to your wallet. The bot never sees a price; it just calls the endpoint.

rotation To rotate, mint a fresh key, swap the one OPENAI_API_KEY line on the bot box, then roger grant revoke <old-name> (or Revoke on the keys page). Revocation takes effect on the next request. A leaked key cannot be re-shown, so rotate rather than recover.

Self-use is $0

When the consuming identity is the node's own owner, billing is $0 automatically. A --self grant extends that to your own headless boxes and agents - the remote-agent-box pattern: run a Hermes or bot box anywhere, set the grant as its key, and it talks to your home GPU free, over the same broker, with no inbound port on either side.

It survives restarts - the always-on path

A grant key is bound to your account (plus any optional model/node scope), not to a session, a process, or a particular share. That is the whole point for an always-on service: the key keeps working across every kind of restart, with nothing to regenerate.

stop sharingthe key stays valid; requests just wait for a station - it is not revoked by going off air
restart sharingyour station re-registers under the same stable node id (the <station>-<model> callsign, persisted - never the hostname/port), so the same key reaches it again with no change on the bot box
restart the botnothing to re-issue - the two env lines are all the bot needs; reboot it whenever
broker redeploysthe node registry is persisted and re-hydrated (§4), so a broker restart does not invalidate the key or your station

Because the broker resolves a grant to you and routes only to your nodes (it intersects the issuing account's nodes with the grant's optional --nodes allow-list, server-side - a grant can never name or reach a node you do not own), a leaked key cannot be pointed at someone else's GPU, and your own services find your GPU again the moment it comes back on air. This is the recommended path for petlings, a Minecraft chat bot, a Ping-style concierge, or any service that should outlive a reboot - mint the key once, set the two env lines, and never touch it again.

Caps, scope, and revoke

A fresh grant gets a conservative daily token cap by default so a forgotten or leaked key is self-limiting. Everything beyond --name and free-vs-priced lives behind --advanced.

--daily-capmax tokens / UTC-day; default 2,000,000 (0 = unlimited)
--monthly-capmax tokens / UTC-month (0 = unlimited)
--models / --nodesrestrict to specific models or to specific nodes of yours
--rpmsustained requests / minute (0 = broker default)
--expireslifetime, e.g. 30d or 2026-12-31 (default: never)

Manage them from the CLI or the dashboard:

$ roger grant list              # your keys + usage per label
$ roger grant show my-bots       # one key's scope, caps, usage
$ roger grant revoke my-bots     # kill it (effective next request)
second guardrail Per-key caps bound one key. Your account-wide monthly spend cap (§6, on the Billing page) bounds what every sponsored key can cost you put together - so even a runaway fleet of priced grants can never exceed the one number you set for the account.

A private band: two boxes, one frequency code

Sometimes you do not want a band on the public dial at all - you just want your inference box and your agent box to find each other. Put the model on air with --private and the radio gives you a one-time frequency code instead of a public listing: a fake MHz frequency plus a short pairing code. The band never appears on the open market; only a box that tunes to that exact frequency can reach it.

# on your inference box (the home GPU)
$ roger share qwen3-coder-30b --private
  private band on air. frequency code (shown ONCE):

      147.520 MHz · 8F3K-9M2Q

  hand this to a box or a friend to tune in.
the code is shown once The frequency code prints exactly once, at share time. Copy it then - the broker keeps only its hash, so a lost code is re-issued, never recovered.

On the box that should use it, tune to the frequency instead of the open band:

# on your agent box (anywhere)
$ roger use qwen3-coder-30b --freq 147.520MHz-8F3K-9M2Q

If both boxes are the same account, self-use is $0 - the private band is just a direct line between two of your own machines, no public signal in between. Hand the code to a friend and they can tune in from their own box: they pay your price per token, or nothing if the band is free.

--privaterequires roger login - a private band is tied to your account
free limit1 private band per account (the broker enforces it: a second mint is refused with "private band limit reached"); more come in $5 packs (Phase 2)
--freqtune a consuming box to a code: the MHz part is cosmetic, the code after it is what matters
vs. grant keysa grant is a bearer API key for many users; a private band is one off-market frequency two boxes share

From inside the app

You do not need the CLI. In [2] SHARE, press h on a model to HIDE it onto a private band; a fresh band drops you on a one-time code card where c copies the frequency code (shown once) and the broker keeps only its hash. Press h again to put the model back on the open market. To consume a private band from the app, tune in with the code: from [1] TUNE IN press ~ (a guarded key, deliberately not f - it pre-seeds the prompt with /freq  so you just paste the code) or run /freq <code> yourself. The band list then shows ONLY that frequency and the header reads FREQ <display> instead of OPEN MARKET; esc (or a bare /freq) clears the tune back to the open market.

§6 · Wallet & billing

One account, one wallet

Your account is your GitHub login, and your account is your wallet. You can browse and open free endpoints with no account; you log in once you want a funded wallet to spend on paid stations. Top-ups run through Stripe Checkout; Stripe is the payments provider and the licensed money transmitter - RogerAI never takes custody of your funds outside Stripe.

$ roger balance                 # your wallet balance in $
$ roger balance --topup 25      # add $25 (opens Stripe Checkout)
$ roger balance --topup         # bare --topup defaults to $10

Spend is debited live, per token, against the wallet balance. A conditional debit on the broker side means a wallet can never go negative. The web dashboard mirrors the same numbers.

The $1 starter credit

The first time you log in, the broker drops a $1 starter credit on your wallet - enough to try a paid model before you ever top up. It is a one-time seed (idempotent per account) and goes to the first 1000 accounts, so claim it early; roger topup adds more after.

Price safety - you will not get surprised

RogerAI guards both sides of the dial so a typo or a hostile listing cannot quietly drain your wallet:

default ceilingif you set no --max-out, the client still caps every connect at $10 / 1M out - this floor applies even with --yes, so an unattended bot can never silently bind to an absurd price
type-the-priceto accept a station above $20 / 1M out you must type the price back, not just press y - a deliberate speed bump on genuinely expensive bands
operator ceilingthe broker refuses to list a public node priced above $100 / 1M out or $50 / 1M in (base price and every scheduled window) - only a typo or a deterrent price trips it; if an operator truly wants an unreachable price they share --private instead (§5)
/accountprofile, linked GitHub, payout status, export / delete
/billingtop-ups and the credit ledger
/usageper-request usage and receipts
/dashboardyour nodes, grants, and live activity

Logging in from the app

Press L in the app for the login panel. Logged out, it explains what a login unlocks (a funded wallet + earning as a provider) and waits - press Enter to start the GitHub device flow (it opens your browser and shows a short code to paste if the browser does not open), esc to back out. The panel never logs you in just by opening it. Once linked, the same L shows you as @your-login with your balance and asks log out? [y/N] - a deliberate confirm, default no. The preset's label flips LOGIN ↔ LOGOUT with your state.

§7 · Earnings & payouts

From served token to your bank

Operator earnings follow a lifecycle: accrued → held → payable → paid. A served request credits your share (70% of the cost). That credit is escrowed - held - long enough to cover the chargeback and fraud-review window, then becomes payable, and you take it out through Stripe Connect.

FIG. 3 - EARNINGS LIFECYCLE 
serve request  →  credits your 70% share
      ↓
   [HELD]      held through the 120-day window (covers the chargeback tail)
      ↓
  [PAYABLE]    hold cleared, no open dispute, Stripe KYC complete
      ↓
   [PAID]      you request a payout → Stripe Transfer → payout to your bank

The policy

Minimum hold120 days from a request's settlement before earnings become payable - the hold itself covers the late-chargeback tail, so no separate reserve is withheld
Minimum payout$25 - below this, transfer and reporting overhead is not worth it
Scheduleyou request; payable balances are released on a monthly batch
KYC gateStripe Connect onboarding (transfers capability active) before any payout

One line: a 120-day hold, a $25 minimum, monthly batched payouts, and mandatory Connect KYC before any payout. There is no separate reserve - the 120-day hold is the protection. Earnings still accrue (to held) for an un-onboarded operator, so supply is not discouraged - they are just not transferable until KYC clears.

Connect onboarding

Payouts use Stripe Connect Express: Stripe hosts the KYC and onboarding UI and pays out to your bank. Start it from the dashboard - the /payouts page shows held (with release dates), payable, lifetime paid, your Connect status, and a Request payout button once you are over the minimum. RogerAI never stores your identity documents; Stripe holds those.

Cash out from the CLI

The whole payout flow is also headless. You must be logged in (roger login); KYC and a GitHub link gate every payout.

$ roger payout                  # same as `payout status`
$ roger payout status           # Connect/KYC state, payable vs held, next-payable date, policy
$ roger payout onboard           # complete Stripe Connect KYC (opens the browser)
$ roger payout request           # request a payout of your payable balance
$ roger payout history           # past payouts and their states

Aliases: payouts / cashout for the command; withdraw for request.

Config file, flags, and env

Config lives at $XDG_CONFIG_HOME/rogerai/config.json (your OS user-config dir). It holds the broker URL, your user handle, saved spend limits, the onboarding state, and the saved share config. Inspect and set it with roger config.

§10 · Verification & trust

Don't trust the station - check it

Every request co-produces a lineage receipt: the node signs {model, in, out, price, prev_hash} and the broker counter-signs it (Ed25519, hash-chained). That gives a non-repudiable, append-only audit trail - you can prove what was billed, by whom, in what order.

Independent token re-count

The completion text flows through the broker, so the broker re-tokenizes the prompt and completion with the canonical tokenizer for the claimed model and bills on its own count, not the node's self-reported usage block. A station cannot inflate token counts to over-charge you.

The active probe

A broker goroutine periodically issues a synthetic job to each node - on admission, before the node's first paid traffic, and then every 10-15 minutes. It measures real TTFT and tok/s independent of organic traffic and gates wrong-model and dead-or-filtered stations out of the band before a paying request can hit them.

How we keep the band honest

in practice RogerAI layers several checks so a station that serves the wrong model, a heavy quant, or pads its token counts is caught and made unprofitable: (a) co-signed lineage receipts on every token, (b) an independent token re-count - we re-count, not just trust the node, and (c) active canary probes that test each station's latency and that it serves the model it advertises - plus logprob spot-checks, a held-earnings escrow with slashing for caught cheating, and an optional TEE-attested confidential tier (§4). Together these make misreporting statistically detectable and not worth it. As with any model provider, verify that outputs meet your quality bar for your use case, and report anything off - we act on reports, and a station's record follows it.

Pre-dispatch safety screen

The broker is the one choke point every prompt passes through, so it is where illegal content (CSAM and the like) is blocked before it ever reaches a provider. Before a prompt is relayed to a station - and on the public Ping concierge on the site - its text is checked against a moderation model. Today that screen runs on Groq's Llama Guard, our moderation provider; the check happens in transit (the prompt is not stored for it) but it does mean the text is read by the screen, and sent to Groq, at dispatch time. That is the one honest exception to the broker being content-blind, and it is spelled out on the privacy page, the Terms, and the Acceptable Use Policy.

Report a station

If a station serves something off - low quality, the wrong model, or abuse - report it. Every station card on the Models page has a report control (category: abuse / CSAM / spam / quality / other, plus an optional request id and free-text detail); a report can be filed anonymously and POSTs to the broker. Enough independent reports auto-eject a node from routing, and a station's record follows it. You can also email abuse@rogerai.fyi directly. Security issues go to security@rogerai.fyi (see the security page and /.well-known/security.txt).

§11 · Troubleshooting / FAQ

When the signal drops

"no local LLM detected" on share

RogerAI tries Ollama, LM Studio, llama.cpp, vLLM, and LiteLLM on their default ports, reads common env vars, and enumerates real listening ports - no brute scan. If your server is on a custom port and not found, point at it: roger share --upstream http://127.0.0.1:8081/v1. Confirm it serves GET /v1/models.

"403" when sharing

A priced node requires a linked GitHub. Either run roger login first, or share for free (drop the price flags) - free shares need no login.

"no station is serving <model> right now"

The band for that model is empty - no station is on air for it this moment. The app says so plainly and points you at the fix: put one on air with [2] (SHARE your own GPU) or tune in [1] to a different band. The same hint shows in the channel and in [0] AGENT if a station fades mid-request; tuning in re-scans the band.

"balance empty" / not logged in

Browsing is free; spending on paid stations needs a funded wallet. Run roger login, then roger balance --topup. Free stations and your own grant keys cost nothing.

"address in use"

use auto-picks a free port from 4141 up, so this is rare. To pin one, pass --port (advanced).

Do I need to open a port to share?

No. The provider agent uses an outbound long-poll; there are no inbound ports, and the broker reaches only your model port - no shell, no files, no context.

Can I point at a different broker?

Yes - federation. roger config set broker <url> or ROGER_BROKER=<url> for one session. Pick who you trust.

Appendix B · The OpenAI-compatible API

Drop-in for anything that speaks OpenAI

When you tune in, the radio runs a local OpenAI-compatible server on 127.0.0.1. Set the base_url to that endpoint and the model param to the band you connected, with the printed key. Any OpenAI SDK or tool works unchanged.

base_urlhttp://127.0.0.1:<port>/v1
api keyroger-local - the local proxy key printed by use / /endpoint (it authorizes the local server, not the broker)
modelthe band id you connected, e.g. qwen3-coder-30b
FIG. 4 - EXAMPLE REQUEST 
$ curl http://127.0.0.1:4141/v1/chat/completions \
    -H "Authorization: Bearer roger-local" \
    -H "Content-Type: application/json" \
    -d '{
      "model": "qwen3-coder-30b",
      "messages": [{"role": "user", "content": "Say roger."}]
    }'

In Python with the OpenAI SDK:

>>> from openai import OpenAI
>>> client = OpenAI(base_url="http://127.0.0.1:4141/v1", api_key="roger-local")
>>> client.chat.completions.create(
...     model="qwen3-coder-30b",
...     messages=[{"role": "user", "content": "Say roger."}],
... )

Roger that. The band is open.

Appendix B2 · Use RogerAI in opencode (and other OpenAI-compatible tools)

Point your editor at the band

roger use <model> runs a local OpenAI-compatible endpoint at http://127.0.0.1:4141/v1 (key roger-local), so any tool that speaks the OpenAI API can route through RogerAI to whatever station you tuned in to - your spend cap, your max-out, and the broker's routing all still apply.

Wire up opencode

Three steps: pin the port so the URL is stable, tell opencode about the provider, then pick the model.

1. Tune in and pin the port:

$ roger use Qwen3-Coder-Next --port 4141

2. Add this to ~/.config/opencode/opencode.json:

{
  "$schema": "https://opencode.ai/config.json",
  "provider": {
    "roger": {
      "npm": "@ai-sdk/openai-compatible",
      "name": "RogerAI (local)",
      "options": {
        "baseURL": "http://127.0.0.1:4141/v1",
        "apiKey": "roger-local"
      },
      "models": {
        "Qwen3-Coder-Next": {
          "name": "Qwen3-Coder-Next (local)",
          "tool_call": true
        }
      }
    }
  }
}

3. In opencode, pick the roger/Qwen3-Coder-Next model. Requests now flow through RogerAI to the station you tuned in to.

Any OpenAI-compatible client

It is always the same three things - a base URL, a key, and a model:

baseURLhttp://127.0.0.1:4141/v1
api keyroger-local
modelthe band you ran roger use with, e.g. Qwen3-Coder-Next

Cursor, Continue, Cline, a raw SDK - they all take those same three values. Two notes: keep roger use running (it is the proxy), and tool_call: true turns on tool / function calling for models that support it.
